Execution governance runtime

One gate. One chain. Every coding agent.

Chitin sits on the tool-call boundary, not inside the model loop. It normalizes actions across Claude Code, Codex CLI, Gemini CLI, Copilot CLI, Hermes, and OpenClaw, evaluates them against one typed policy, and writes every decision to a tamper-evident chain.

Stable primitives around unstable agent substrates: gate, chain, bounds, and signals. The kernel authorizes. Drivers adapt. The chain remembers.

What ships

Lean by design

Chitin is not another orchestrator, agent framework, or approval system. It is the execution-governance layer that can sit beneath those systems and impose one runtime contract on all of them.

Thesis

The moat is cross-driver normalization plus typed policy enforcement plus a hash-linked decision chain. Everything else is instrumentation around that kernel.

Demo

The gate sees the action class, not just the shell string. A forced push, recursive delete, governance self-modification, or remote code execution attempt is blocked before the side effect lands.

Policy surface

26 live invariants on the public surface

This page enumerates the first 26 active rule categories from the current shipped policy surface: the deny rules that define the boundary and the baseline allows that keep work moving.

  1. 1. No recursive deleteBlocks `file.recursive_delete` outright.
  2. 2. No destructive `rm -rf`Catches direct shell-based mass deletion.
  3. 3. No execute-code delete bypassBlocks recursive delete expressed through embedded interpreters.
  4. 4. No force pushRejects rewritten shared history.
  5. 5. No commit to protected branchRequires work to start on a feature branch.
  6. 6. No protected pushStops direct push to `main`, `master`, or unresolved bare push targets.
  7. 7. No `.env` writeAgents do not mutate secret-bearing env files.
  8. 8. No system path writeBlocks writes and deletes under protected OS paths.
  9. 9. No credential path writeProtects `~/.ssh`, cloud creds, shell init, and similar state.
  10. 10. No shell write into protected pathsCatches `tee`, `dd`, and redirect-style bypasses.
  11. 11. No governance self-mod via shellPrevents inline interpreter writes to policy and plugin paths.
  12. 12. No governance self-modificationAgents cannot rewrite their own guardrails.
  13. 13. No `terraform destroy`Live infrastructure teardown is denied.
  14. 14. No remote code execBlocks fetch-and-execute shapes like `curl | bash`.
  15. 15. Hermes cannot spawn frontier CLIs directlyDispatch goes through the substrate workflow, not ad hoc shell calls.
  16. 16. Reads are safe by default`file.read` is explicitly allowed.
  17. 17. Git read operations are safe`diff`, `log`, `status`, and worktree listing stay open.
  18. 18. GitHub read operations are safeIssue, PR, and API reads are allowed.
  19. 19. GitHub write operations are boundedCreate, close, merge, and PR creation are allowed, with bounds still enforced where relevant.
  20. 20. Non-push git ops are allowedCommit, checkout, merge, and worktree management stay available.
  21. 21. Non-protected push is allowedFeature-branch push remains the default path.
  22. 22. Test execution is allowedRunning tests should never require special pleading.
  23. 23. Delegation is allowedSubagent dispatch remains available as a governed action.
  24. 24. HTTP requests are allowedNetwork reads stay available unless a tighter rule overrides them.
  25. 25. MCP calls are allowedTool surfaces exposed through MCP remain first-class governed actions.
  26. 26. Shell execution is allowed by defaultThe shell is open except for the explicit dangerous classes above.
