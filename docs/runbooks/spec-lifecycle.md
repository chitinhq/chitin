# Spec lifecycle runbook

How to write spec front-matter, when to transition status, and how
the index stays in sync.

Schema defined in
[`docs/superpowers/specs/2026-05-13-spec-lifecycle-metadata.md`](../superpowers/specs/2026-05-13-spec-lifecycle-metadata.md).

## Front-matter schema

Every file under `docs/superpowers/specs/**/*.md` (except `INDEX.md`)
starts with a YAML front-matter block:

```yaml
---
status: open
owner: claude-code
kanban: t_5f50f6a8
implementation_pr: null
superseded_by: null
effective_from: 2026-05-13
effective_to: null
---
```

All seven fields are required. `null` is a valid value for
`kanban`, `implementation_pr`, `superseded_by`, and `effective_to`.

### Field semantics

| Field | Meaning |
|---|---|
| `status` | Lifecycle state. Closed enum — see below. |
| `owner` | Who maintains the spec. Convention: `claude-code`, `jared`, `red`, `jared+claude`, `red+claude`. |
| `kanban` | Source kanban ticket id (e.g. `t_5f50f6a8`), or `null` if no ticket. |
| `implementation_pr` | Integer PR number that implemented the spec, or `null` if not yet implemented. Required when `status: implemented`. |
| `superseded_by` | Path to the spec that replaced this one, or `null`. Required when `status: superseded`. |
| `effective_from` | Date the spec entered its current lifecycle (`YYYY-MM-DD`). |
| `effective_to` | Date the spec became implemented or superseded, or `null` if still active. |

Extra fields are allowed (e.g. `reason`, `audience`, `purpose` on
older superseded specs). The linter ignores them.

## Status enum

| Value | When to use |
|---|---|
| `draft` | Being written; not yet ready for implementation work. |
| `open` | Accepted; implementation-eligible. New specs land here. |
| `implemented` | Shipped. `implementation_pr` MUST be set. |
| `amended` | Live, but the body has post-spec amendments above it. |
| `superseded` | Replaced. `superseded_by` MUST point to an existing spec path. |

The linter (`scripts/check-spec-frontmatter.py`) enforces:

- `status` is one of the five values above.
- `status: implemented` → `implementation_pr` is non-null.
- `status: superseded` → `superseded_by` points to an existing file.

## Transitions

The lifecycle is operator-driven; no daemon promotes spec status
automatically. The transitions are:

```
   draft ──→ open ──→ implemented
             │            │
             │            └─→ (terminal until superseded)
             │
             └─→ amended ──→ superseded
                  │              ▲
                  └──────────────┘
```

### draft → open

When the spec is ready for implementation work. Update:

```yaml
status: open
effective_from: <today>
```

### open → implemented

When the spec's Done-condition is met and the implementation PR is
merged. Update:

```yaml
status: implemented
implementation_pr: <PR number>
effective_to: <merge date>
```

### open → amended

When the spec's body has been updated post-acceptance to reflect
what actually shipped or to add scope. Add an `Amendment Log`
section near the top of the spec body. Update:

```yaml
status: amended
```

`effective_to` stays `null` because the spec is still live.

### any → superseded

When a later spec replaces this one. The file stays in
`docs/superpowers/specs/` (the legacy `/superseded/` subdir was retired
in the 2026-05-13 doc purge). Update:

```yaml
status: superseded
superseded_by: docs/superpowers/specs/<later-spec>.md
effective_to: <date of supersession>
```

The original `effective_from` stays unchanged — it records when the
spec was first written. Operators may delete fully-retired specs after
review; INDEX.md is regenerated either way.

## INDEX.md regeneration

`docs/superpowers/specs/INDEX.md` is auto-generated by
`scripts/regen-spec-index.py`. Regenerate after any change to a
spec's front-matter:

```bash
python3 scripts/regen-spec-index.py
```

CI runs `python3 scripts/regen-spec-index.py --check` and fails the
build if the committed `INDEX.md` disagrees with the regenerated
output. Commit the index alongside your spec change.

## Adding a new spec

1. Create `docs/superpowers/specs/YYYY-MM-DD-<slug>.md`.
2. Add the front-matter block. `effective_from` is today; `status`
   is `draft` if you're still writing, `open` if it's ready for
   implementation.
3. Write the body.
4. Run `python3 scripts/check-spec-frontmatter.py` to validate.
5. Run `python3 scripts/regen-spec-index.py` to refresh the index.
6. Commit both files (spec + INDEX.md) in the same PR.

## Migrating an old spec to the new schema

Use the retrofit pass from
`docs/superpowers/specs/2026-05-13-spec-lifecycle-metadata.md` as a
reference. For ad-hoc updates: open the file, add the
seven-field block at the top, infer `effective_from` from the
filename, and set `status` from your knowledge of the spec's
current state. Run the linter to confirm.

## FAQ

**Why is `kanban` allowed to be null?**

Older specs (April 2026 and earlier) predate the kanban-linked
workflow. They have no source ticket. Leave `kanban: null` rather
than fabricating an id.

**Can a spec have multiple PRs?**

`implementation_pr` is a single int. If a spec is implemented
across multiple PRs, set it to the final PR and reference the
others in the body (e.g. in an "Implementation Log" section). The
field is for index lookup, not exhaustive history.

**What's the difference between `amended` and `open`?**

`open` means the spec body is the single contract. `amended` means
the body still describes the intent but the top of the spec has an
Amendment Log section explaining what changed post-acceptance.
Readers should look at amendments before assuming the body is
fully current.
