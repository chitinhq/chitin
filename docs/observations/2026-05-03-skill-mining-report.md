# Skill-mining report (semantic verbs)

Generated by `python -m analysis.skill_mine` over `~/.chitin/events-*.jsonl`.

## Triage

Three actionable patterns surfaced from this run:

### Pattern 1 — swarm worktree orientation (HIGHEST FREQUENCY)

n-grams: `ls → shell-chain → shell-chain`, `find → ls`, `cat → ls`, `read-ts → find`, `read-ts → shell-chain`.

Every dispatched swarm agent starts with 5–10 probes against its `swarm-worktrees/swarm-X` dir (ls -la, find, cat README.md, ls node_modules/@nx/, etc).

**Verdict: determinism wins.** Have the dispatcher write a `WORKTREE_INDEX.md` to each worktree at creation: project layout, key paths, test entry points. Agent reads one file instead of probing for ten minutes.

### Pattern 2 — Nx generator scaffolding by hand

n-gram: `cat → mkdir → edit-json → edit-json → edit-json` (3 sessions, also as 4-gram and 5-gram extensions).

Sessions: `swarm-nx-generator-spec-plan-doc`, `swarm-scheduler-rank-ingest-notify`, `swarm-slack-l1-notifier`, `swarm-slack-l2-actions`, `swarm-nx-generator-workspace-lib`. Agents read a test fixture, then write project.json + tsconfig.json + package.json by hand.

**Verdict: determinism wins.** This is what Nx generators are FOR. Three of those entries are literally `nx-generator-*` — agents are scaffolding the generators they're building because the generators don't exist yet. The pattern should disappear once those PRs (#229, #238, #248) merge. Worth re-running this analysis post-merge to confirm.

### Pattern 3 — `shell-chain` is the dominant verb (144 of 1159 decisions ≈ 12%)

Sample: `ls ~/.cache/chitin/ && ls ~/.cache/chitin/swarm-runs/ || ls ~/.cache/chitin/swarm-state/`.

Agents reach for `&&`/`||` chains constantly — cheap state probing.

**Verdict: skill OR determinism.** A `/state` skill that runs a curated probe set, or better, a persistent state file the dispatcher updates. Lower priority than 1+2.

## Method

- Group events from ALL `events-*.jsonl` by `chain_id` (the canonical session id) before mining. On this box, each `events-<run_id>.jsonl` file happens to map 1:1 to one chain_id, so the per-file analysis I shipped initially produced the same result. The grouping by chain_id is for robustness on environments where one session spans multiple run files (e.g., subagent dispatch, repeated hook invocations within one Claude session). Verified post-fix: 89 chain_ids across 89 files, none spanning.
- Per session, extract ordered sequence of canonical "verbs" (action_type + abstracted target).
- shell.exec target → first command + first non-flag arg (`gh pr view` → `gh-pr-view`); chains collapse to `shell-chain`.
- file.read/write → `read-<ext>` / `edit-<ext>`.
- Enumerate n-grams n=2..5; filter trivial repeats (all-same-verb, all-read, all-edit).
- Score = sessions × n. Surfaces patterns repeated across many sessions AND of substantial length.
