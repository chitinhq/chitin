# Amendment v1.1 — Operator unlock recovery without process restart

**Amends**: [spec.md](../spec.md) (v1.0 — implemented in PR #923, merged 2026-05-23)
**Created**: 2026-05-23
**Status**: Draft (design only — no implementation yet)
**Branch (when implemented)**: `feat/091-v1.1-unlock-recovery`
**Trigger**: Operator-reported failure mode observed 2026-05-23 via clawta in #clawta — see "Observed failure" below.

## Why this amendment exists

Spec 091 v1.0 lands a sticky in-process `stopHookActive[sessionId]` flag in the chitin-governance openclaw plugin. Once `continue:false` fires for a session, *every* subsequent `before_tool_call` for that session short-circuits to `{block:true, stop:true}` until the openclaw process is restarted (declared at `apps/openclaw-plugin-governance/src/index.mjs:19`, checked at `:78-84`, set at `:116-126`; `sessionId` built at `:67`).

The v1.0 design depends on a contract noted in the same file at line 74:

> openclaw's harness will keep calling before_tool_call (it does not consume the `stop` field — R1 Case B)

In production this combines into the failure mode described below: a session that hit one `continue:false` becomes permanently quiescent for the entire process lifetime, and the only operator recovery is an out-of-band `openclaw session delete` plus restart. Edits to chitin policy (`chitin.yaml`) or `~/.chitin/gov.db` cannot reach the openclaw process's in-memory `stopHookActive` map.

This amendment **does not relax the v1.0 hard-stop contract**. A `continue:false` deny still terminates the session's tool-call activity instantly. What changes is the **recovery path**: an explicit operator gesture (an "unlock", recorded durably) clears the in-memory flag on the next `before_tool_call`, without requiring an openclaw process restart.

## Observed failure (2026-05-23)

Operator (Jared) observed clawta unresponsive in #clawta. Clawta self-diagnosed via Discord message:

> The stop signal is intercepting before any tool can execute. This is a session-level issue — the current session is poisoned with the stop state. The unlock you did in gov.db should take effect in the *next* session. This one is stuck.

Independent verification on the operator box:

| Check | Value | Interpretation |
|---|---|---|
| `sqlite3 ~/.chitin/gov.db "SELECT locked FROM agent_state WHERE agent IN ('openclaw','openclaw-plugin','main')"` | all `0` | Kernel-side lockdown counter is clear; the agent IS unlocked from chitin's perspective. |
| `stopHookActive[sessionId]` in the openclaw plugin process | `true` (inferred from sticky block behavior) | Plugin-side in-memory flag is the actual block; cannot be cleared without a process restart. |
| Spec 091 `data-model.md` Entity 3, "Cleared when" column | `plugin reload (process restart)` | This is the intended v1.0 behavior. |

The mismatch is intentional in v1.0 but problematic in practice. The kernel/db is the operator's source of truth for "is this agent locked"; the plugin's in-memory flag should be downstream of that, not authoritative.

## Goals

- **G1**: An operator can clear a session's sticky stop-hook via a single, durable, chitin-native gesture (writing to gov.db or invoking a kernel subcommand). The next `before_tool_call` for that session evaluates normally.
- **G2**: No openclaw process restart required. No file system sentinel files in `/tmp` or similar. No Discord-side workarounds.
- **G3**: The v1.0 hard-stop contract is preserved unchanged. A `continue:false` deny still terminates immediately; only the *clearing* path is new.
- **G4**: The unlock gesture leaves a chain-event audit trail (operator X cleared session Y at time T, with reason Z) so the lockdown→unlock sequence is forensically reconstructable.

## Non-goals

- Auto-clearing on a timer/TTL. Rejected because it conflicts with v1.0's "terminate, don't negotiate" stance and risks masking real upstream-harness bugs.
- Per-rule or per-tool granularity (e.g., "unlock for rule X only"). v1.1 unlocks the whole session; finer-grained unlocks can land in a later amendment if needed.
- Changes to the openclaw harness or upstream npm package. The fix stays inside chitin (kernel + plugin).
- Changes to `chitin.yaml` policy semantics. The unlock is operator state, not policy.

## Dependency — kernel session-state surface (spec 096)

The kernel-side CLI subcommands, schema additions, and chain event types this amendment depends on are owned by **[spec 096 — operator session-state surface](../../096-operator-session-state-surface/spec.md)**. Spec 096 must merge before this amendment's implementation begins.

From spec 096 this amendment consumes:

- `chitin-kernel session status -agent <id>` — JSON returning `{agent, locked, locked_ts, unlock_ts, lock_epoch, total, level}`. The plugin uses this to detect operator unlocks.
- `chitin-kernel session unlock -agent <id> [-reason <text>]` — the operator gesture that clears `agent_state.locked` and advances `lock_epoch`. This amendment does not invoke it directly; operators do.
- `agent_state.lock_epoch` — a monotonically-increasing per-agent integer that lets this amendment detect transitions via epoch comparison rather than timestamp arithmetic.
- `session_unlocked` chain event — emitted by the kernel when an unlock lands; the plugin does not produce this event, only observes its downstream effect via `session status`.

Anything not listed above (the kernel subcommand semantics, the schema migration, the operator-facing UX, the chain event payload schemas) is NOT in this amendment's scope — see spec 096.

## New functional requirements (amendment scope)

These extend spec 091 v1.0 on the **plugin-consumer side only**. IDs use the `AFR-` prefix to keep them distinct from v1.0's `FR-001..FR-009` and avoid conflicting with `speckit-lint`'s FR numbering check.

- **AFR-001 (Source of truth)**: The kernel's `~/.chitin/gov.db` (as exposed by spec 096's `session status` subcommand) MUST own the per-agent unlock state. The plugin's in-memory `stopHookActive[sessionId]` is downstream of `gov.db` — it can never be "more locked" than `gov.db` says the agent is. A `continue:false` from the kernel can still locally pin the in-memory flag, but the operator's unlock (observed via `lock_epoch` advance) clears it.

- **AFR-002 (Lock-epoch instrumentation on set)**: When the plugin sets `stopHookActive[sessionId] = true`, it MUST also call `chitin-kernel session status -agent <agentId>` to capture the current `lock_epoch` value, and store it in a sibling `stopHookActiveEpoch: Map<string, number>`. This is the comparison anchor for AFR-003. If the status query fails at lock-set time, the plugin MUST still set the sticky flag (fail-closed); the cached epoch is then `null`, and AFR-003's comparison defaults to "any future `locked: false` response clears the flag".

- **AFR-003 (Unlock detection on short-circuit)**: When the plugin's `before_tool_call` finds `stopHookActive[sessionId] === true`, it MUST consult `chitin-kernel session status -agent <agentId>` before returning the sticky block. The plugin MUST clear its in-memory flag (and the `forcedContinuations` counter for that session) and proceed to normal evaluation if **either** of the following holds: (a) the kernel response carries `locked: false`, **OR** (b) the kernel response carries `lock_epoch > stopHookActiveEpoch[sessionId]` (operator re-locked then re-unlocked in between; the new generation is what matters). If both `locked: true` and `lock_epoch === stopHookActiveEpoch[sessionId]`, the sticky block stands.

- **AFR-004 (Plugin-side unlock chain event)**: When the plugin clears its in-memory flag in response to an observed kernel unlock (AFR-003), it MUST emit a `stop_hook_cleared` chain event via the kernel's existing `emit` subprocess path, with payload `{session_id, agent, cleared_at, kernel_lock_epoch}`. This is the plugin-side acknowledgment that distinguishes "the kernel unlocked" (logged via spec 096's `session_unlocked`) from "the plugin actually noticed and resumed" (logged via this event). Fire-and-forget — telemetry failure must not block the deny→allow transition. Mirrors `stop_signal_ignored` (FR-009).

- **AFR-005 (Fail-closed on kernel query error)**: If `chitin-kernel session status` fails (timeout, kernel binary missing, malformed response) while the plugin is checking for an unlock, the plugin MUST keep `stopHookActive` set and return the sticky block. Absence of a positive unlock signal is treated as "stay stopped". Mirrors v1.0 FR-003.

- **AFR-006 (Lock-set authority preserved)**: The plugin NEVER writes to `gov.db` or invokes `chitin-kernel session unlock` / `session lock` directly. Lock-set is the kernel's domain (automatic via `RecordActionDenial`, or operator via spec 096's CLI). Constitution §1 preserved.

- **AFR-007 (No regression in soft-block path)**: The unlock-check (AFR-003) is invoked ONLY when `stopHookActive[sessionId]` is already true. For soft-deny sessions (where the flag was never set), behavior is byte-identical to v1.0 — no per-call kernel status query. The plugin does not query the kernel for status on every `before_tool_call`; only on the short-circuit path.

- **AFR-008 (Caching to prevent burst hammering)**: The plugin MAY cache the kernel's `session status` response for up to 1 second per agent, so an agent making burst tool calls during a sticky-stop state does not generate one subprocess per call. The cache MUST be invalidated when the plugin clears its flag (AFR-003), so the next call after recovery does not see a stale "locked" reading. Implementation may skip caching for v1.1 if the burst-call concern doesn't materialize in practice; this requirement is "MAY", not "MUST".

## New success criteria

- **ASC-001 (One-shot operator recovery)**: With clawta in a sticky-stopped state (`stopHookActive[sessionId] === true`, the symptom observed 2026-05-23), the operator runs `chitin-kernel session unlock -agent <agent-id>` (subcommand owned by spec 096) and within one minute, the next tool call clawta attempts is gated normally (allow or deny, but no longer short-circuits). No openclaw restart required.

- **ASC-002 (Audit trail intact)**: For the recovery in ASC-001, the chain contains, in order: (a) the original `continue:false` deny event, (b) any forced-continuation block events, (c) a `session_unlocked` event from the kernel (spec 096), (d) a `stop_hook_cleared` event from the plugin (this amendment, AFR-004), (e) the normal gate decision for the next tool call. `jq` over `~/.chitin/events-*.jsonl` reconstructs the sequence.

- **ASC-003 (No regression in v1.0 contracts)**: SC-001 through SC-004 from v1.0 still pass. In particular: a single `continue:false` deny still produces exactly one deny event followed by a session-close (or stop) state; lockdown_loop_detected stays at zero; soft denies still allow agent retries.

- **ASC-004 (Fail-closed verified)**: With the kernel binary deliberately renamed (simulating AFR-005's failure path), an operator-issued unlock has no effect — the plugin stays sticky-stopped. Restoring the kernel and re-issuing the unlock recovers the session.

- **ASC-005 (Lock-epoch monotonicity respected)**: A scenario where the kernel re-locks the agent (via auto-escalation or operator `session lock`) AFTER an unlock the plugin already noticed — and then unlocks again — produces TWO `stop_hook_cleared` events from the plugin (one per epoch advance the plugin observed and acted on). The plugin never gets "stuck" on a stale epoch.

## Design sketch (handed to /speckit-plan)

This is not the implementation plan — it's the seed for the next planning pass.

### Data model deltas

Extend Entity 3 (Plugin reentrancy state) from `data-model.md`:

```js
const stopHookActive       = new Map();   // sessionId → boolean  (v1.0, unchanged)
const stopHookActiveEpoch  = new Map();   // sessionId → number|null  (v1.1, NEW — AFR-002; captured at lock-set time)
const forcedContinuations  = new Map();   // sessionId → number  (v1.0, unchanged)
```

The kernel-side data model deltas (`agent_state.unlock_ts`, `agent_state.lock_epoch`, `session_unlocked` chain event, `session_locked` chain event) are owned by [spec 096](../../096-operator-session-state-surface/spec.md) — not duplicated here.

Add Entity 6 (plugin `stop_hook_cleared` chain event) — mirrors the existing `stop_signal_ignored` shape from v1.0 FR-009. Payload: `{event_type:"stop_hook_cleared", session_id, agent, cleared_at, kernel_lock_epoch}`.

### Sequence — operator recovery (new happy path)

```
Pre-state: stopHookActive[sid] = true; stopHookActiveEpoch[sid] = 5
            (plugin captured epoch=5 from `session status` at the moment it set sticky-stop)
1. Operator runs (spec 096):
     chitin-kernel session unlock -agent clawta -reason "policy relaxed"
2. Kernel writes gov.db (spec 096): agent_state.locked = 0, unlock_ts = T1, lock_epoch = 6
3. Kernel emits chain event (spec 096):
     { event_type: "session_unlocked", agent: "clawta", ts: T1, lock_epoch_after: 6, reason: ... }
4. Agent (clawta) attempts a tool call later (at T2 > T1)
5. Plugin before_tool_call sees stopHookActive[sid] = true → enters unlock check (AFR-003)
6. Plugin spawns (spec 096 interface):
     chitin-kernel session status -agent clawta
     → { locked: false, lock_epoch: 6, unlock_ts: T1, ... }
7. Plugin compares (AFR-003):
     - locked: false → clear  OR
     - lock_epoch 6 > cached 5 → clear
     Either condition matches → proceed to clear path.
8. Plugin clears stopHookActive[sid], stopHookActiveEpoch[sid], forcedContinuations[sid]
9. Plugin emits chain event (AFR-004):
     stop_hook_cleared { session_id, agent, cleared_at: T2, kernel_lock_epoch: 6 }
10. Plugin proceeds to normal evaluateRouter / evaluateHookGate path for the original tool call.
    The original tool call is gated against the (now-relaxed) policy; allow or deny is the
    normal outcome — no sticky short-circuit.
```

### Sequence — fail-closed (AFR-005)

```
Pre-state: stopHookActive[sid] = true
1. Plugin enters unlock check; spawns chitin-kernel session status -agent clawta
2. Spawn fails (kernel binary missing) OR times out
3. Plugin treats as "no unlock signal" → returns the v1.0 sticky block
4. log.warn captures the failure for operator visibility
```

### Risks / open questions (for /speckit-plan to resolve)

- **OQ1**: Per-call kernel subprocess cost. Today's gate call is already one subprocess per `before_tool_call`; the AFR-003 status query adds another only on the short-circuit path (which by definition the agent shouldn't be hitting many times). Bound it: if `stopHookActive` is true, the agent is in a degenerate state — paying ~10ms for a status query per call is acceptable until they stop trying. AFR-008 makes the optional cache explicit.
- **OQ2**: AFR-002 captures `lock_epoch` at the moment the plugin sets `stopHookActive`. If the kernel's auto-escalation lockdown happens concurrently with the plugin's `continue:false` handling (a race where lock_epoch advances mid-flight), the cached epoch may be stale by one. AFR-003's two-condition clear (locked:false OR epoch advance) handles this correctly — the next status response will show the new epoch, and the plugin will clear. Worst case: one extra sticky-block returned before recovery. Acceptable.
- **OQ3**: How does this interact with spec 092 (codify-swarm-orchestrator) if the orchestrator gains its own session-state surface? Spec 096's surface is single-host gov.db; if 092 introduces a cross-host session model, the plugin may need to consume both. Investigate during /speckit-plan.
- **OQ4**: Multi-agent sessions: `sessionId = `openclaw-${agentId}-${pid}` so the agent dimension is already keyed. Unlock at the agent level (`agent_state.agent`) is the right granularity for now. Revisit if openclaw ever multiplexes agents within one process.

## What's deliberately out of scope for v1.1

- **Kernel CLI + schema + chain event definitions**: owned by [spec 096](../../096-operator-session-state-surface/spec.md). This amendment is the *consumer*; spec 096 is the *producer*. Land spec 096 first.
- **Spec 092 alignment**: if the codified swarm orchestrator (spec 092) lands its own session-state surface, this amendment may need re-anchoring. Tracked as OQ3.
- **Upstream openclaw fix**: making the harness honor `stop:true` is still the right long-term fix (R1 in v1.0 research.md). v1.1 makes the chitin side recoverable without that upstream change; the upstream PR remains a separate work-unit.
- **A timer/TTL fallback**: rejected per non-goals. If the operator never unlocks, the session stays stopped — which is the correct degraded state, not an outage.
